# Running the Dataset Pipeline on the Janelia Cluster

This guide explains how to run the 24-keypoint 3D pose dataset generation pipeline
on the Janelia LSF compute cluster. It is written for someone with no prior cluster
experience. After following this guide, you will be able to generate the full
rory + wilfred dataset in ~5–8 hours instead of ~7 days on a single workstation.

---

## Prerequisites

You will need:
- A Janelia cluster account (contact IT to get one, billed to `johnson`)
- SSH access to the cluster login node (`login1`)
- Access to the Johnson lab NAS at `10.10.1.28` as user `ratan`
- The code copied to your cluster home (see below)

---

## 1. Cluster overview

The Janelia cluster uses **LSF** (IBM Platform Load Sharing Facility) as its job
scheduler — not SLURM. Key commands:

| Command | Purpose |
|---|---|
| `bsub < job.lsf` | Submit a job script |
| `bjobs` | List your running/pending jobs |
| `bkill <jobid>` | Cancel a job |
| `bqueues` | List available queues and their status |
| `bhosts` | List compute nodes |

Jobs are billed per CPU-hour. GPU jobs (`gpu_a100`) cost **$0.10/hr**.

Available GPU queues (as of 2026-03):

| Queue | GPU | Notes |
|---|---|---|
| `gpu_a100` | NVIDIA A100 | Best availability, recommended |
| `gpu_gh200` | NVIDIA GH200 | Faster, fewer slots |
| `gpu_l4` | NVIDIA L4 | Very busy (62k jobs pending), avoid |
| `gpu_t4` | NVIDIA T4 | Older, slower |

## 2. One-time SSH setup

### 2a. Generate an SSH key pair (if you don't have one)

On your **local machine**:
```bash
ssh-keygen -t ed25519 -C "your_name@janelia.org"
# Press enter to accept defaults (no passphrase needed for automation)
cat ~/.ssh/id_ed25519.pub
```

### 2b. Add your local key to the cluster

Your key needs to be in `~/.ssh/authorized_keys` on the cluster. The cluster home
is mounted at `/mnt/johnsonlab/` on the local workstation, so you can do this
without logging in:

```bash
# On the local workstation (as user with write access to /mnt/johnsonlab):
mkdir -p /mnt/johnsonlab/.ssh
echo "$(cat ~/.ssh/id_ed25519.pub)" >> /mnt/johnsonlab/.ssh/authorized_keys
chmod 700 /mnt/johnsonlab/.ssh
chmod 600 /mnt/johnsonlab/.ssh/authorized_keys
```

Then test:
```bash
ssh doq@login1 "echo connected"
```

### 2c. Give the cluster access to the NAS

The NAS (`ratan@10.10.1.28`) holds all the video data. The cluster nodes need
SSH access to rsync files from it.

1. Generate a key on the cluster login node:
```bash
ssh doq@login1
ssh-keygen -t ed25519 -N '' -f ~/.ssh/id_ed25519
cat ~/.ssh/id_ed25519.pub
```

2. Add that public key to the NAS. Log into the NAS (web UI at `http://10.10.1.28`,
   or ask the lab manager) and append the key to `~/.ssh/authorized_keys` for the
   `ratan` user. Or if you have password SSH access to the NAS:
```bash
ssh ratan@10.10.1.28
mkdir -p ~/.ssh && chmod 700 ~/.ssh
echo "ssh-ed25519 AAAA...the_key_you_copied..." >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```

3. Test from the cluster:
```bash
ssh doq@login1 "ssh ratan@10.10.1.28 'echo NAS_OK'"
# Should print: NAS_OK
```

### 2d. Add the cluster to your local SSH known_hosts

```bash
ssh-keyscan login1 >> ~/.ssh/known_hosts
```

## 3. Copying code and models to the cluster

The cluster home (`/groups/johnson/home/doq/`) is mounted locally at
`/mnt/johnsonlab/`. You can copy files directly via this mount — no `scp` needed.

### Code
```bash
rsync -av --exclude='dataset/predictions_raw/' \
          --exclude='dataset/rory.csv' \
          --exclude='dataset/wilfred.csv' \
          --exclude='dataset/*.log' \
          --exclude='__pycache__/' \
          --exclude='trt-models/' \
          /home/user/src/analyzeMiceTrajectory/ \
          /mnt/johnsonlab/analyzeMiceTrajectory/
```

### JARVIS models

The cluster already has `~/JARVIS-HybridNet/` with `mouseJan30`. You need to add:

- `mouseClimb4` CenterDetect weights (~90 MB)
- `mouseHybrid24` composite project (~100 MB total; no TRT — those get recompiled on the cluster)

```bash
# mouseClimb4 CenterDetect
rsync -av /home/user/src/JARVIS-HybridNet/projects/mouseClimb4/models/CenterDetect/ \
          /mnt/johnsonlab/JARVIS-HybridNet/projects/mouseClimb4/models/CenterDetect/

# mouseHybrid24 (excludes locally-compiled TRT models)
rsync -av --exclude='trt-models/' \
          /home/user/src/JARVIS-HybridNet/projects/mouseHybrid24/ \
          /mnt/johnsonlab/JARVIS-HybridNet/projects/mouseHybrid24/
```

### Trials CSVs

The trial window files (door-open → door-close frame ranges) must be on the cluster:
```bash
rsync -av /tmp/rory_trials.csv /tmp/wilfred_trials.csv \
          /mnt/johnsonlab/analyzeMiceTrajectory/dataset/
```

These are generated by `cbot_climb_log/extract_trial_frames.py` — re-run that
script if you add new sessions.

## 4. How the cluster pipeline works

### Why a job array?

The dataset has 26 sessions (rory: 17, wilfred: 9), each independent. We run one
LSF job per session: 26 jobs × 1 GPU each = all 26 sessions in parallel.
Total wall time drops from ~7 days (single GPU) to ~5–8 hours.

### Why rsync to /scratch?

The NAS video data (24 TB total, ~800 GB/session) is too large to copy to
`/groups/johnson/` (only 3.8 TB free). Instead, each job:

1. **rsync** the one session it needs from the NAS to `/scratch` (local NVMe per
   node, 1.8 TB). Takes ~15–30 min for ~800 GB at ~400 MB/s LAN.
2. **Run inference** on `/scratch` data (~5–8 hrs at ~15 fps on A100).
3. **Write results** to `/groups/johnson/home/doq/analyzeMiceTrajectory/dataset/`
   (small CSVs, ~20 MB/trial).
4. **Clean up** `/scratch`.

### TRT compilation

TRT (TensorRT) models are compiled for a specific GPU architecture. The local
workstation's TRT models will not work on the cluster's A100/GH200. Each job
checks whether `trt-models/predict2D/centerDetect.pt` exists and, if not,
runs `compile_trt_hybrid24.py` to compile them. This only happens once on the
first job that runs on each node.

## 5. Submitting jobs

### First time: log into the cluster and check the environment

```bash
ssh doq@login1

# Check conda env has the right packages
~/miniconda3/bin/conda run -n jarvis_repro python -c "import torch, cv2; print(torch.__version__)"

# Verify NAS access
ssh ratan@10.10.1.28 "ls /mouse2/climb/rory/ | head -3"

# Verify JARVIS loads
~/miniconda3/bin/conda run -n jarvis_repro python -c "
import sys; sys.path.insert(0, '/groups/johnson/home/doq/JARVIS-HybridNet')
from jarvis.config.project_manager import ProjectManager
p = ProjectManager(); p.load('mouseHybrid24'); print('OK')
"
```

### Test run (one small session before committing to all 26)

```bash
# Session 10 = rory/2026_01_02_14_31_43 (~26k frames, ~107 GB, ~1.5 hrs total)
bsub -J 'dataset[10]' < ~/analyzeMiceTrajectory/cluster/run_session.lsf
bjobs   # watch status: PEND → RUN
```

Monitor the log:
```bash
# Job ID is printed by bsub, e.g. Job <148784102>
tail -f ~/analyzeMiceTrajectory/cluster/logs/job_148784102_10.out
```

### Full run (all 26 sessions)

Use the provided submit script — it shows a confirmation prompt before submitting:
```bash
bash ~/analyzeMiceTrajectory/cluster/submit.sh
```

To submit a subset:
```bash
bash ~/analyzeMiceTrajectory/cluster/submit.sh 11-26   # skip already-done sessions
```

### Monitor progress

```bash
# Job statuses
bjobs -u doq

# Live output from all jobs
tail -f ~/analyzeMiceTrajectory/cluster/logs/job_*.out

# Count finished trials
find ~/analyzeMiceTrajectory/dataset/predictions_raw -name data3D.csv | wc -l

# Check for failures
grep -l FAILED ~/analyzeMiceTrajectory/cluster/logs/job_*.out 2>/dev/null

# See fps achieved on completed sessions
grep 'fps' ~/analyzeMiceTrajectory/cluster/logs/job_*.out
```

## 6. After jobs complete: generate the clean dataset

Once all 26 jobs are done (or any subset), run `filter_dataset.py` to apply the
arena-mask filter and produce the clean per-animal CSVs:

```bash
# On the local workstation (has the arena_mask.npz and calib_params):
python3 /home/user/src/analyzeMiceTrajectory/filter_dataset.py

# Or on the cluster (copy arena_mask.npz there first):
rsync -av /home/user/src/analyzeMiceTrajectory/predictions3D/arena_mask.npz \
          /mnt/johnsonlab/analyzeMiceTrajectory/predictions3D/
ssh doq@login1 "~/miniconda3/bin/conda run -n jarvis_repro \
    python3 ~/analyzeMiceTrajectory/filter_dataset.py"
```

Output: `dataset/rory.csv` and `dataset/wilfred.csv` — 96-column CSVs
(24 keypoints × x, y, z, confidence) with arena-gated frames only.

## 7. Cost estimate

At $0.10/hr per GPU job with 12 CPU cores:

| Sessions | Est. wall time/job | Est. total cost |
|---|---|---|
| 26 (full) | 5–8 hrs | ~$13–21 |
| 26 with stride=3 (60fps) | ~2–3 hrs | ~$5–8 |

Cost is per-job, all running in parallel, so total wall-clock time = longest single job.

To use stride=3 (60fps effective), edit `run_session.lsf` and add `--stride 3` to
the `make_dataset.py` invocation.

## 9. One-time environment setup (already done for `doq`)

The `jarvis_repro` conda environment originally shipped with PyTorch 1.10.1+cu113
(2021). This version is too old for `torch_tensorrt` 2.x. The following upgrade
was applied once on the login node:

```bash
# Upgrade PyTorch to 2.5.1 (matches local workstation)
~/miniconda3/envs/jarvis_repro/bin/pip install \
    torch==2.5.1 torchvision==0.20.1 \
    --index-url https://download.pytorch.org/whl/cu121

# Install TensorRT
~/miniconda3/envs/jarvis_repro/bin/pip install torch_tensorrt==2.5.0
```

If you are setting up a **new cluster account**, run these commands once on the
login node before submitting any jobs. JARVIS itself (`jarvis-mocap 0.1.1`) is
compatible with PyTorch 2.5.1 — no other changes needed.

> Note: `import torch_tensorrt` will print a warning on the login node
> ("Unable to read CUDA capable devices") because the login node has no GPU.
> This is harmless — it works correctly on GPU compute nodes.

## 10. Troubleshooting

**Job stays PEND for a long time**
`gpu_a100` is often busy (check with `bqueues`). Use `gpu_t4` for immediate
slots (older GPU but x86_64 and always available):
```bash
bsub -J 'dataset[10]' -q gpu_t4 < ~/analyzeMiceTrajectory/cluster/run_session.lsf
```
**Do NOT use `gpu_gh200`** — GH200 is an ARM-based Grace Hopper node. The
`jarvis_repro` conda env is compiled for x86_64 and will fail with
`Exec format error` on ARM nodes.

**rsync fails: Permission denied**
The cluster's SSH key is not in the NAS `authorized_keys`. Re-do step 2c above.
Test with: `ssh doq@login1 "ssh ratan@10.10.1.28 'echo ok'"`.

**`/scratch` permission denied**
`/scratch` has per-user subdirectories. If `/scratch/doq` doesn't exist,
the job script will try to create it. If that fails, it falls back to `/tmp`
(fine for sessions ≤700 GB, but slower). To pre-create it:
```bash
ssh doq@login1 "bsub -W 00:05 -n 1 -gpu 'num=1' -q gpu_t4 -P johnson -Is \
    /bin/bash -c 'mkdir -p /scratch/doq && echo done'"
```

**JARVIS project not found**
`make_dataset.py` auto-detects JARVIS at `~/JARVIS-HybridNet/`. Confirm:
```bash
ssh doq@login1 "ls ~/JARVIS-HybridNet/projects/mouseHybrid24/config.yaml"
```

**TRT compilation fails or is slow**
TRT models are compiled fresh on first job per node (takes ~5–10 min).
Compiled models are saved to `~/JARVIS-HybridNet/projects/mouseHybrid24/trt-models/predict2D/`
on the shared NFS home, so subsequent jobs on any node reuse them.
To compile manually (e.g. to pre-warm before full array run):
```bash
bsub -W 01:00 -n 4 -gpu "num=1" -q gpu_t4 -P johnson -Is /bin/bash
# inside the job:
export PATH="$HOME/miniconda3/envs/jarvis_repro/bin:$PATH"
python3 ~/analyzeMiceTrajectory/compile_trt_hybrid24.py
```

**`conda run` / `conda activate` fails in job**
These do not work reliably in non-interactive LSF jobs. The job script
activates the env by prepending its `bin/` to `PATH` directly — do not
change this to `conda activate` or `conda run`.

**Out of scratch space**
`/scratch` is per-node local NVMe (~1.8 TB). A session is ~800 GB, so one job
fits. If a previous crashed job left data: `rm -rf /scratch/doq/job_*`.

**Jobs wrote partial results then crashed**
`--skip-done` checks for `data3D.csv` per trial. A partially-written CSV will
be skipped. To force re-run a specific trial, delete its folder:
```bash
rm -rf ~/analyzeMiceTrajectory/dataset/predictions_raw/rory_SESSION/trial_XXXX_*/
```
Then resubmit just that session:
```bash
bsub -J 'dataset[10]' -q gpu_t4 < ~/analyzeMiceTrajectory/cluster/run_session.lsf
```

**Python type hint error: `unsupported operand type(s) for |`**
Happens if the Python in your env is < 3.10. The codebase uses `X | None`
return type hints which require Python 3.10+. These have been removed from
`make_dataset.py` and `filter_dataset.py` for Python 3.9 compatibility.
If you see this on a new script, remove the `-> Type | None` annotation.
